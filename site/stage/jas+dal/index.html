Concurrency

ÂµFork is driven entirely by asynchronous events. There are no blocking operations. Instruction execution is efficiently interleaved, so all ongoing activities make progress concurrently.

Read more... Activities are extremely cheap to create. In fact, a new activity is created to handle the processing of each event. An activity ends when the effects of the event are either committed (releasing new events) or aborted.

ÂµFork supports dynamic memory allocation. A garbage-collector runs concurrently with all other activities, automatically reclaiming unreachable storage.

Safety

Activities are strongly isolated from each other. State-change is always local to a particular activity. Interaction occurs by generating asynchronous events. This simple design uses less logic and consumes less power to maintain coherency and prevent conflicts.

Read more... ÂµFork has strong memory-safety. There are three primitive types of data:
  • Bounded Integers
  • Pointers to immutable data
  • Addresses for events
There is no conversion between these types and no pointer/address arithmetic.

ÂµFork enforces fine-grained quotas on memory, instructions, and events. These quotas can be used to prevent resource-exhaustion attacks.

Security

ÂµFork implements object-capability security. There is only one kind of capability, to generate an asynchronous event for a specific address. This model can be summarized as, "If you don't have it, you can't use it."

Read more... You have to know an address to generate an event for that address. The address-reference graph is the access-control graph. As mentioned under "Safety", an address cannot be created from an Integer or Pointer.

An address is also opaque. It can be used to generate an asynchronous event, but it cannot be used to read memory. Addresses may be communicated by events.

Introduction

ÂµFork is a novel machine architecture based on the Actor Model of computation. Actors are stateful objects that react to message-events. Each actor has a unique address. Each event designates the address of the actor to which it will be delivered. An actor handles events one-at-a-time, accumulating a set of effects which include:

  • creating new actors (create)
  • creating new message-events (send)
  • defining the actor's future behavior (become)
Event handling ends with either a commit or abort. If the actor commits, the accumulated effects are applied to the system. If the actor aborts, the effects are discarded.

All data values in ÂµFork (including actor addresses) are immutable. This makes them safe to share in message-events. The only mutation allowed is actor behavior replacement (become), which designates the code and data that will be used to handle future message-events.

ÂµFork is a realization of the ideas described in "Memory Safety Simplifies Microprocessor Design", which explains the motivation and rationale in more detail.

The quickest way to get started using ÂµFork is to use the online playground. You can start writing and running ÂµFork programs immediately without any installation. You also use the ÂµFork debugger visualize your program running on the ÂµFork virtual machine and to debug it.
